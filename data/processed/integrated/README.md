# data/processed/integrated

Parquet snapshots produced by `notebooks/02_integration.ipynb` (Phase 2,
silver layer). The two tables here apply the PROVENANCE-deferred
preprocessing the bronze layer left untouched, canonicalise join keys
across sources, and join the resulting frames into the analytical
shapes Phase 4 visual components consume. They depend on the bronze
parquets in the sibling directories under `data/processed/`. This
directory is gitignored and regenerated by re-running the integration
notebook on a fresh kernel.

## models_enriched.parquet

The grain is one row per Jegham model × query length combination — 194
rows, with the Jegham `DataSnapshotOct26` (the frozen October 2026
snapshot) as the spine. The live `artificialanalysis_environmental`
feed and the monthly time-series file are not consumed by this table;
only the frozen snapshot is. Epoch training metadata (`Publication
date`, `Organization`, `Training compute (FLOP)`, `Parameters`,
`Training hardware`, `Training power draw (W)`) is joined where the
model name matches after case-insensitive parenthetical-stripping, with
the Epoch row deduplicated to the most recent `Publication date` per
match key.

This table feeds Component 1 (per-query energy, water, and carbon for 
hosted frontier LLMs), Component 2 (model-comparison panel
— training-side markers), Component 4 (training compute trajectory)
 , Component 5 (aggregate scaling to 1M/1B/100B prompts),
  and Component 7 (white-hat / black-hat statics).
   Jegham supplies the per-query numbers; Epoch supplies the
training-side context.

Two caveats are surfaced in the notebook and carried into the table
rather than swept aside. Twenty-one of sixty-six unique Jegham models
are unmatched in Epoch — typically because they post-date Epoch's
snapshot (the Claude 4 family is the canonical example) or because
their hosted-variant naming pattern does not collapse cleanly to an
Epoch entry even after stripping parens. Those rows are preserved with
NaN Epoch columns rather than dropped. Separately, `Training power
draw (W)` is empty across all 134 matched rows: Epoch tracks this
column primarily for older research-era architectures (CNNs, BERT-class)
where labs published wall-clock GPU power figures, and does not
estimate it for the hosted frontier LLMs Jegham covers. The column is
preserved in the schema but Components 2 and 4 use `Training compute
(FLOP)` and `Parameters` as their primary training-side axes.

## regions_geographic.parquet

The grain is one row per cloud region — US rows at state grain via
two-letter postal codes, non-US rows at country grain via ISO-3 alpha
codes. The table has 37 rows: 10 AWS regions and 27 Azure regions, with
each provider's synthetic aggregate row excluded.

AWS and Microsoft multiplier files supply region, WUE, and CIF; the
rows are enriched with Ember grid carbon intensity at the most recent
year available per region, with Aqueduct baseline water stress from the
`bws` indicator at `Tot` weighting, and with the IM3 atlas US facility
footprint (campus-layer count and total square footage) aggregated to
state level. Component 3 (geographic map) consumes this table directly;
Components 5 and 7 use it for geographic context. Per the Phase 2
design decision, this table provides geographic context only — it does
**not** override Jegham's pre-computed per-query numbers in Component 1
by recombining `Jegham energy × multiplier CIF/WUE`.

Two regions return NaN for `water_stress_score`: HKG (Hong Kong, rolled
under mainland China by Aqueduct's hydrological-basin methodology, with
no Hong Kong-specific country row) and SGP (Singapore, which lacks an
agricultural watershed and returns Aqueduct's `-9999` sentinel, coerced
to NaN at ingestion). These values are kept as NaN rather than
substituted by adjacency; Phase 4 visualisations render them as "no
data" tiles, and the report's Limitations section names both cases
explicitly.

Detailed column-level documentation lives in
`notebooks/02_integration.ipynb` itself — the `validate_and_write`
helper's `must_have_columns` parameter and the section markdown cells
together document every column — not in this file.
